# 产品需求文档 (PRD)

## 商品精灵 - 电商多平台智能商品上架与管理工具

> **文档版本**: v1.0  
> **创建时间**: 2026-05-02  
> **痛点来源**: 知乎、今日头条、小红书用户痛点采集

## 1. 项目背景

### 1.1 痛点发现

通过WebSearch从知乎、今日头条、小红书等平台采集用户痛点数据，发现**电商多平台商品上架与管理重复劳动**是电商从业者最迫切需要解决的问题。

**典型用户反馈**：
- *"双11期间团队3人专职处理多平台订单，仍因信息不同步导致超卖退款，直接损失超10万"* —— 服装电商老板
- *"单SKU多平台上架平均耗时25分钟，参数错误占商品管理问题的42%"* —— 电商服务商调研
- *"每天后台、各类工具导出的数据堆积如山，却始终难以真正洞察"* —— 电商运营

### 1.2 市场规模

- **受影响用户**: 800万+ 中小电商卖家、电商代运营、品牌方电商团队
- **付费意愿**: 极高（现有ERP系统年费数千至数万，用户仍大量采购）
- **痛点频次**: 每天（日均上架/管理商品10-100个SKU的卖家占比超70%）

### 1.3 竞品分析

| 竞品 | 优势 | 不足 | 年费 |
|------|------|------|------|
| 金蝶星辰 | 功能全面、品牌背书 | 价格高(万元+/年)、学习成本大 | 1万-5万 |
| 班牛 | 电商场景深入 | 中小卖家用不起 | 5000-3万 |
| 有赞 | 生态完善 | 偏平台型、非效率工具 | 6000-2万 |
| 店长管家 | 价格亲民 | 功能较基础 | 1000-5000 |

**核心差异化机会**: 现有ERP系统**功能重、价格高、学习成本大**，中小卖家急需一款**轻量、低价、开箱即用**的商品上架效率工具。

## 3. 核心用户故事

### US-1: 一次录入多平台上架
> 作为一名在淘宝、拼多多、抖音三平台经营的卖家，我希望录入一次商品信息后，系统能自动适配各平台的标题规则、主图尺寸、规格参数，一键上架到所有平台，这样我可以把每个SKU的上架时间从25分钟缩短到3分钟。

### US-2: 智能库存同步
> 作为电商运营，我希望系统能实时同步各平台的库存数据，当某平台售出商品后自动扣减其他平台库存，避免超卖导致的退款损失。

### US-3: 批量商品管理
> 作为服装类目卖家，我每季上新100+款SKU，每款5-10个颜色尺码。我希望能通过Excel批量导入商品信息，系统自动拆分为各平台格式并批量上架。

### US-4: 多平台数据汇总
> 作为店铺老板，我希望每天打开一个看板就能看到所有平台的销售数据、库存状态、异常订单，而不需要逐个平台后台查看。

## 5. 主流程

### 5.1 核心业务流程

```
用户登录 → 录入商品信息(或Excel批量导入)
    ↓
选择目标平台(多选) → 平台规则引擎自动适配
    ↓
预览各平台效果(标题/主图/规格/价格) → 确认/微调
    ↓
一键上架 → 上架状态追踪
    ↓
库存自动同步 → 异常告警
    ↓
数据看板查看 → 经营分析
```

### 5.2 平台规则适配流程

```
输入: 标准化商品信息 + 目标平台列表
    ↓
平台规则匹配:
  - 淘宝: 标题≤60字(30个汉字), 主图800×800, 规格(颜色/尺码)
  - 拼多多: 标题≤60字, 主图750×1000, 规格(规格名/规格值)
  - 抖音: 标题≤30字, 主图800×800, 短视频封面
  - 京东: 标题≤30字, 主图800×800, 参数(品牌/型号/材质)
  - 快手: 标题≤30字, 主图800×800
    ↓
智能处理:
  - 标题: 关键词提取 + 平台SEO优化 + 字数截断
  - 图片: 自动裁剪/缩放为平台要求尺寸
  - 价格: 自动换算(含税/不含税/平台佣金)
  - 规格: 属性映射(颜色→颜色分类, 尺码→尺码分类)
    ↓
生成各平台上架数据 → 用户预览确认
```

## 8. 异常场景与边界条件

| 场景 | 处理策略 |
|------|----------|
| 平台API限流 | 自动排队重试，分批处理，避免触发风控 |
| 平台授权过期 | 提前7天提醒续期，过期后暂停同步 |
| 上架失败(标题违规等) | 记录原因，标记异常，通知用户修改 |
| 图片格式/尺寸不符 | 自动裁剪缩放，无法处理时提示用户 |
| 库存同步冲突 | 最后写入胜出策略 + 异常告警 |
| 批量导入数据错误 | 逐行校验，错误行标记并生成错误报告 |
| 价格同步异常 | 价格变动超20%时暂停同步并人工确认 |
| 网络异常 | 本地缓存操作记录，恢复后自动重试 |

## 9. 技术方案概要

### 9.1 技术选型

| 层级 | 技术方案 | 理由 |
|------|----------|------|
| 前端 | Next.js + React + Ant Design | 企业级UI组件、表格/表单能力强 |
| 后端 | Python FastAPI | 数据处理能力强、生态丰富 |
| 数据库 | PostgreSQL + Redis | 关系数据 + 缓存/消息队列 |
| 消息队列 | Redis Streams / RabbitMQ | 异步任务处理（上架/同步） |
| 图片处理 | Sharp (Node.js) | 高性能图片裁剪缩放 |
| 部署 | Vercel (前端) + Railway (后端) | 低成本、自动扩缩容 |

### 9.2 系统架构

```
[Web前端] → [API网关] → [商品服务] → [规则引擎]
                              ↓
                         [适配服务]
                              ↓
                    ┌─────────┼─────────┐
               [淘宝API] [拼多多API] [抖音API] [京东API] [快手API]
                              ↓
                         [库存同步服务] → [消息队列]
                              ↓
                         [数据采集服务] → [数据看板]
```
